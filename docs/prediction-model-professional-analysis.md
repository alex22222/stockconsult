## 二、模型矩阵详细分析

### 2.1 本地 sklearn 集成模型 (核心)

**入口**: `main.py` → `model_trainer.py`

**模型组合**（加权投票）：

| 模型 | 权重 | 状态 | 评注 |
|------|:--:|:--:|------|
| GradientBoosting | 0.25 | ✅ 启用 | 树模型主力，max_depth=3，较保守 |
| RandomForest | 0.25 | ✅ 启用 | 集成多样性来源 |
| ExtraTrees | 0.20 | ✅ 启用 | 更多随机性，防过拟合 |
| LogisticRegression | 0.10 | ✅ 启用 | 线性基线 |
| AdaBoost | 0.10 | ✅ 启用 | 对异常值敏感 |
| SVM_RBF | 0.05 | ❌ 禁用 | "高维下易过拟合" |
| MLP | 0.05 | ❌ 禁用 | "高维下易过拟合" |

**训练方式**：
- 时间序列分割 (TimeSeriesSplit, 3折)
- 滚动训练窗口：60天 (train_window)
- 每模型单独做 SelectKBest 特征选择 (top 30)
- 标准化严格只在训练集 fit

**优点**：
- 数据泄露防护意识好：scaler.fit 只在训练段、特征选择只在训练段、交叉验证每 fold 单独 fit
- 集成多样性不错：树模型 + 线性模型 + boosting 家族覆盖
- 模型按股票代码隔离存储

**问题**：
- 5个启用的模型中，3个是树模型变体（GradientBoosting/RandomForest/ExtraTrees），高度同源，集成多样性实际有限
- 权重固定（配置写死），进化引擎虽有 `evolve_model_weights` 但不影响实时预测
- 预测输出只有二分类（涨/跌），不预测幅度或概率分布

### 2.2 云函数轻量预测模型

**入口**: `index_lightweight.py` → `predict_stock()`

**四因子线性评分**：

| 因子 | 权重 | 信号来源 |
|------|:--:|------|
| 趋势 (Trend) | 0.35 | MA5/MA10/MA20 排列 |
| 动量 (Momentum) | 0.25 | 近期涨跌幅 + RSI 修正 |
| 量能 (Volume) | 0.20 | 成交量趋势 + 价格方向 |
| 技术 (Technical) | 0.20 | MACD + 布林带 + RSI |

**评分逻辑**：每个因子输出 0-100 分，加权求和得到综合分数，映射为涨跌概率。

**优点**：
- 纯 Python 标准库，零依赖部署
- 可解释性强，每步计算可追踪
- 响应快，适合云函数

**问题**：
- **这不是真正的机器学习模型**——它是硬编码的规则评分器
- 权重是拍脑袋定的（如 "trend_score=85 当多头排列时"），没有经过任何统计验证
- 4个因子全部来自价格数据，本质上在做同一件事的不同表述
- 分数截断（5%-95%）掩盖了真实不确定性

### 2.3 回归预测模型

**入口**: `regression_model_trainer.py` → `regression_backtest.py`

**模型**：GradientBoostingRegressor, RandomForestRegressor, Ridge, ElasticNet

**目标**：预测5日收益率（百分比），而非涨跌方向

**策略**：预测收益 > threshold（如2%）时才买入

**优点**：
- 从分类转向回归是正确方向——能区分"涨0.5%"和"涨5%"
- 回测对比了多个基准（买入持有、MA5/20、随机信号）
- 包含止损止盈和仓位管理

**问题**：
- 实际运行结果疑似不佳（代码中有 but 未见输出留存）
- R² 值未见报告
- 5日预测周期与 T+1 制度匹配度仍需验证

### 2.4 LSTM 深度学习模型

**入口**: `lstm_601318.py`

**架构**：LSTM(2层, hidden=64) + Attention + FC → Sigmoid

**目标**：预测中国平安次日涨跌

**问题**：
- 仅用于单只股票（601318），未推广
- 从代码风格和结果看，更像是实验性探索
- 深度学习在小样本（~190有效样本）上的过拟合风险极高
- PyTorch 依赖意味着无法部署到云函数

### 2.5 异常波动分类器

**入口**: `anomaly_classifier.py`

**目标**：预测未来5日是否出现异常波动（|收益率|>2%），四分类

**问题**：
- 未集成到主流程中
- 未见回测结果
- 类别不平衡问题（大涨/大跌是稀有事件）未做专门处理（仅用了 class_weight='balanced'）

### 2.6 进化引擎

**入口**: `evolution_engine.py`

**功能**：
- 基于近期准确率自动调整因子权重和模型权重
- 特征选择进化（保留 top-K 重要特征）
- 引入探索噪声防止陷入局部最优

**评价**：
- 思路正确——自适应权重是量化领域的常见做法
- 但**在模型本身没有 alpha 的情况下，进化引擎只是在优化噪声**
- 如果基础信号没有预测力，优化权重分配不会创造预测力

## 三、特征工程评估

### 3.1 特征全景

系统有两套特征构建路径：

**全量特征** (`build_features`, ~100+维)：
- 市场环境 (8维)：指数趋势、波动率、相关性
- 大盘能量 (12维)：成交量比率、动量、换手率
- 市场情绪 (8维)：恐惧贪婪指数、连续涨跌、振幅
- 技术指标 (40+维)：MACD、KDJ、RSI、布林带、均线、OBV、ATR、K线形态
- 板块热度 (6维)：相对强弱、板块资金
- 资金异动 (10维)：成交量异常、主力资金
- 美股因子 (5维)：QQQ/DIA/SPY/FXI 涨跌幅 + 综合评分
- 扩展宏观 (8维)：北向资金、涨跌停、国债收益率
- 交互特征 (4维)：MACD×RSI、换手率×涨跌幅等
- 滞后特征 (12维)：T-2/T-3 关键指标

**精简特征** (`build_compact_features`, ~16维)：
- 动量 (3维)、波动率 (2维)、量能 (2维)、技术位置 (3维)
- 趋势状态 (3维)、大盘相对 (2维)、K线形态 (2维)
- 非价格宏观 (6维)：北向资金、美股隔夜、涨跌停比、国债收益率

### 3.2 特征质量评估

**正面**：
- 特征类别覆盖广，从微观到宏观都有涉及
- 精简特征路径是明智的——减少噪声
- 有 `feature_evaluator.py` 做 IC 分析和跨股票稳定性检验
- 有 `compare_features.py` 做特征对比

**负面**：
- **全量特征中 80%+ 是价格/成交量的数学变换**
  - MACD 的各种变体、均线交叉信号、布林带位置、RSI 散度...
  - 这些特征共享相同的底层信息源（OHLCV），彼此高度相关
  - 学术界普遍认为纯技术指标在 A 股的预测力已基本被套利殆尽
- **真正独立的信息源极少**：
  - 北向资金（可能有用）
  - 隔夜美股（对中国 ADR 相关股票有传导，但 A 股整体相关性弱）
  - 涨跌停统计（反映市场热度）
  - 国债收益率（宏观环境）
  - 估值因子 PE/PB（仅 paper_trading_5day 中用到）
- **特征/样本比严重失衡**：100+ 特征 / ~190 有效样本 ≈ 1:1.9
  - 理想比例应 > 1:10
  - 这意味模型几乎一定能找到"伪模式"
- **IC 筛选结果暗示信号极弱**：
  - `build_compact_features` 中 `ic_filter=True` 只保留 8 个特征：
    `mom_20d, price_vs_ma20, us_overnight_score, price_pctile_20d, consecutive_down, ma5_above_ma20, mom_5d, consecutive_up`
  - 除了 `us_overnight_score`，其余全是价格衍生特征
  - 且这些特征的最佳 IC 可能也仅在 0.03-0.05 范围

### 3.3 缺失的关键特征类别

| 缺失类别 | 重要性 | 可获取性 |
|----------|:--:|:--:|
| 新闻/NLP 情绪 | ⭐⭐⭐⭐⭐ | investoday 可用 |
| 综合评分 (investoday score) | ⭐⭐⭐⭐ | investoday 可用 |
| 机构持仓变化 | ⭐⭐⭐⭐ | investoday/季报 |
| 分析师一致预期 | ⭐⭐⭐ | investoday 研报 |
| 行业景气度指标 | ⭐⭐⭐ | 需要外部数据 |
| 供应链/另类数据 | ⭐⭐ | 获取成本高 |
| 股东增减持 | ⭐⭐⭐ | investoday 可用 |
| 融资融券余额 | ⭐⭐⭐ | investoday 可用 |

## 四、回测与验证体系评估

### 4.1 回测引擎矩阵

| 引擎 | 文件 | 预测目标 | 持有期 | 评估指标 |
|------|------|----------|:--:|------|
| 基础回测 | backtest_evaluator.py | 次日涨跌 | 1天 | 收益/夏普/回撤/准确率 |
| 滚动回测 | backtest_engine.py | 次日涨跌 | 1天 | 同上 + 对比基准 |
| 回归回测 | regression_backtest.py | 5日收益 | 5天 | 同上 + 胜率/均收益 |
| 周度回测 | backtest_weekly.py | 5日收益 | 5天 | 同上 |
| 快速回测 | backtest_fast.py | 次日涨跌 | 1天 | 同上 |
| V2回测 | backtest_v2.py | 次日涨跌 | 1天 | 同上 |
| 模拟盘 | paper_trading_5day.py | 5日收益 | 5天 | 实际盈亏 |

**问题**：6个回测引擎并行存在，功能高度重叠，各自使用略微不同的参数和假设。这种碎片化意味着没有一个"权威"的回测结果可以作为决策依据。

### 4.2 回测方法论审查

**做得正确的**：
- ✅ 滚动窗口训练，训练集始终在测试集之前
- ✅ 标准化器只在训练集 fit
- ✅ 特征选择器只在训练集 fit
- ✅ 包含交易成本建模（佣金0.025%、印花税0.1%、滑点0.1-0.2%）
- ✅ 最低佣金5元的约束

**存在严重问题的**：

1. **T+1 时间错配未解决**
   - 回测假设 T 日收盘生成信号，T+1 日收盘价执行
   - 实际上 T+1 日买入当天无法卖出
   - 预测 T+1 收盘 vs T 收盘的涨跌，但实际盈亏取决于 T+2 开盘 vs T+1 开盘

2. **涨跌停未建模**
   - 涨停日发出买入信号 → 实际买不到
   - 跌停日发出卖出信号 → 实际卖不掉
   - 回测中这些信号被当作"可执行"

3. **除权除息未处理**
   - 分红送股导致的价格跳空会被模型当作真实涨跌

4. **回测结果几乎全部缺失**
   - 6个回测引擎都有代码，但都没有留下实际运行结果
   - `paper_trading_5day.py` 中露笑科技因"回测负收益"被禁用
   - 这是唯一可观察到的实际回测信号——且是负面的

### 4.3 评估体系 (model_evaluation.py)

**设计质量**：⭐⭐⭐⭐⭐ （框架设计专业）

包含7大评估维度：
1. 统计显著性（二项检验 + 置换检验）
2. 基准对比（随机/买入持有/趋势/反向）
3. 置信度校准（ECE + Brier Score）
4. 过拟合检测（样本内外差距）
5. 经济意义（扣除成本后净收益 + 信息比率）
6. 模型稳定性（滚动窗口 CV + 趋势检测）
7. 游程检验（错误是否随机分布）

**实际使用**：⭐ （从未在真实数据上运行）

`__main__` 块只测试了合成数据：
```python
np.random.seed(42)
n = 200
y_true = np.random.choice([0, 1], size=n, p=[0.48, 0.52])
```

这套评估体系的代码质量是这个项目中最好的部分，但它从未被用于评估真实模型。

### 4.4 预测跟踪现状

从 `public/data/prediction_*.json` 和 `prediction_tracker.py` 来看：
- **历史预测记录几乎为零**——每只股票仅 0-1 条记录
- **零条已验证**——从未在 T+1 收盘后回填实际结果
- **准确率统计为空**——系统运行了，但没有闭环

## 五、当前模型的核心缺陷

### 🔴 致命缺陷（不解决则模型无实用价值）

#### 5.1 没有证明预测能力

这是最根本的问题。在 100+ 个源代码文件中：

- **没有一份可查阅的回测结果报告**
- **没有一条已验证的预测跟踪记录**
- **已知的唯一回测信号是负面的**（露笑科技因回测亏损被禁用）
- **评估体系从未在真实数据上运行过**

一个交易系统最基础的要求是"预测准确率统计显著高于 50%"，而本项目尚未证明这一点。

#### 5.2 日频预测在 A 股不可执行

```
模型视角：T日收盘→信号→T+1收盘→验证
现实约束：T日收盘→信号→T+1开盘买入→T+2开盘卖出（最早）
```

这两个随机变量不同。回测中模拟的理想执行与现实存在系统性偏差。这个偏差足够大，足以让回测中微利的策略在实盘中变为亏损。

#### 5.3 特征同源性和过拟合

- 100+ 特征中 80%+ 是 OHLCV 的数学变换
- 特征/样本比 ~1:1.9（理想 > 1:10）
- 启用的 5 个模型中 3 个是同源的树模型
- 极度缺乏独立信号源（新闻情绪、机构行为、基本面因子）

#### 5.4 数据与执行断层

当前状态是"离线研究，手动运行"：
- 数据靠 `local_data_provider.py` 从本地 CSV 读取
- 预测生成需要人工执行 Python 脚本
- 没有定时任务自动触发
- 云函数代码存在但未在生产环境中持续运行
- 前端显示预测但无历史准确率

### 🟡 严重缺陷（影响实用性）

#### 5.5 风控体系不完整

| 缺失项 | 影响 |
|--------|------|
| 仓位分级管理 | 只有满仓/空仓两种状态 |
| 市场状态识别 | 牛/熊/震荡市用同一策略 |
| 组合层面风控 | 单股票独立交易，无行业分散、无相关性控制 |
| 最大回撤熔断 | 连续亏损后无暂停机制 |

#### 5.6 目标函数混乱

系统内同时存在：
- 分类目标：`model_trainer.py` 预测次日涨跌 (1/0)
- 回归目标：`regression_model_trainer.py` 预测 5 日收益率
- 评分目标：`index_lightweight.py` 四因子综合评分
- 异常目标：`anomaly_classifier.py` 预测大涨/大跌

六套回测引擎各自独立，没有统一的评估标准和真实的对比结论。

#### 5.7 集成策略的虚假多样性

5 个启用的模型（GBDT, RF, ExtraTrees, LR, AdaBoost）中：
- 3 个是决策树集成变体，高度相关
- LR 是线性模型，在非线性特征空间中表现受限
- AdaBoost 对噪声敏感，在小样本下容易过拟合
- SVM 和 MLP 因过拟合被禁用——说明作者意识到了问题，但没有解决根本原因（特征太多/样本太少）

### 🟢 次要缺陷（影响效率但不致命）

#### 5.8 代码冗余

- 6 个回测引擎，功能高度重叠
- 2 个 `feature_engineer.py` 的 `build_features` 函数（全量 + 精简）
- 2 套预测管线（本地 sklearn + 云函数线性规则）
- 多个 `backtest_*.py` 的 main 函数各写一套打印格式

#### 5.9 技术债务

- 配置硬编码在两处（`config.py` 和 `paper_trading_5day.py` 的 `STOCK_CONFIG`）
- 路径拼接使用字符串拼接而非 `pathlib`
- 部分文件缺少类型注解
- `requirements.txt` 可能不完整（LSTM 需要 torch）

## 七、改进路线图

### Phase 0：证明或证伪（1-2周）⭐⭐⭐⭐⭐ **当前阶段**

这是必须先完成的步骤。在投入更多开发之前，需要诚实地回答一个问题：

**这个模型比扔硬币好吗？**

具体行动：
1. **运行完整回测**
   - 4只股票，每只至少250个交易日
   - 输出：总收益、年化收益、夏普比率、最大回撤、胜率、盈亏比
   - 对比基准：买入持有、MA5/20金叉、随机入场
   - 使用 `regression_backtest.py` 的对比框架

2. **启动每日预测跟踪**
   - 每个交易日收盘后自动运行预测脚本
   - T+1/T+5 收盘后自动回填实际结果
   - 至少积累 30 个交易日后做二项检验

3. **运行 model_evaluation 综合评估**
   - 在真实股票数据上跑 7 项检验
   - 输出综合评分（0-100）
   - 如果评分 < 40 → 模型不具备预测能力，进入 Phase 1 重新设计

### Phase 1：修复根本问题（2-4周）

**假设 Phase 0 确认模型有微弱预测力（准确率 53-57%）**：

4. **解决 T+1 执行错配**
   - 将预测标签从"T+1 收盘 vs T 收盘"改为"T+2 开盘 vs T+1 开盘"
   - 或统一迁移到 5 日持有期策略（与 paper_trading_5day 对齐）
   - 推荐后者：5 日周期降低了交易频率，减少了 T+1 的影响

5. **降维 + 去冗余**
   - 使用 `feature_evaluator.py` 的跨股票 IC 分析结果
   - 保留 IC 绝对值 > 0.03 且跨股票方向一致的特征
   - 目标：将特征数从 100+ 降至 15-25

6. **引入独立信号源**（最重要）
   - 接入 investoday MCP API 获取：
     - `get_stock_score` — 综合评分（技术/情绪/基本面/行业四维度）
     - `list_entity_related_news` — 新闻情绪 sentimentScore
     - `get_stock_finance_valuation` — 估值排名（PE/PB 行业分位）
     - `get_stock_finance_profit_ability` — 盈利排名（ROE/毛利率行业分位）
   - 这些是当前系统最缺乏的**独立 alpha 来源**

7. **统一目标函数**
   - 选择 5 日预测作为唯一目标（与模拟盘一致）
   - 废弃日频分类模型或将其降级为辅助信号

### Phase 2：建立风控体系（2-3周）

8. **仓位管理**
   - 实现分级仓位：置信度 > 0.6 → 60%仓位，> 0.7 → 80%，< 0.55 → 不交易
   - 凯利公式：根据 rolling 窗口的胜率和盈亏比动态计算最优仓位

9. **市场状态识别**
   - 基于沪深300 60日均线斜率判断牛/熊/震荡
   - 熊市降低仓位上限至 30%，震荡市降低交易频率

10. **A 股特殊规则适配**
    - 涨停日跳过买入信号
    - 跌停日跳过卖出信号
    - 在回测中标记不可执行信号并单独统计

### Phase 3：工程化（2-4周）

11. **建立自动化流水线**
    - 定时任务（crontab/GitHub Actions）：每日 15:30 拉取数据 → 生成信号
    - 预测结果自动写入 JSON，前端自动展示
    - T+1/T+5 自动验证，准确率实时更新

12. **统一回测框架**
    - 合并 6 个回测引擎为一个
    - 支持多股票、多策略并行回测
    - 输出标准化报告

### Phase 4：进阶（Phase 0-3 全部通过后）

13. 组合层面风控（协方差矩阵、行业集中度、最大回撤熔断）
14. 模拟盘 3 个月验证（在模拟环境中跑完一个完整牛熊周期）
15. 小资金实盘测试（5000-10000 元，严格纪律）
16. 接入更多另类数据（融资融券、龙虎榜、大宗交易）

## 八、下一步最重要的行动

综合以上分析，**按优先级排序的最重要行动**：

### #1 🚨 跑通回测并公开结果（最重要）

**为什么这是第一优先**：在知道模型是否有效之前，所有其他改进都是盲目的。

具体操作：
```bash
cd cloudfunctions/stock-predictor
python regression_backtest.py  # 已有对比框架，直接运行
```

如果结果表明所有策略都不如买入持有，或者夏普 < 0.5，那么：
- **不要再往模型架构上投入时间**
- **核心问题不是模型不够好，而是当前的特征集根本不包含 alpha**
- **下一步应该转向引入独立信号源（investoday API），而非优化现有模型**

### #2 📊 启动每日预测跟踪闭环

在 `scripts/daily_pipeline.sh` 基础上，增加：
- 定时触发（crontab 每个交易日 15:30）
- 自动验证前一日预测（对比实际收盘价）
- 将验证结果写入 JSON，前端实时展示
- 目标：30 个交易日后，有一份可引用的准确率数据

### #3 🔌 接入 investoday 独立信号源

当前系统最致命的问题不是模型架构，而是**特征中几乎没有独立 alpha**。

investoday ¥12.9/3月 的体验包可以解锁：
- 综合评分（多维度 alpha）
- 新闻情绪（NLP alpha）
- 估值排名（价值因子 alpha）
- 盈利排名（质量因子 alpha）

这些数据已被顶级机构验证，且是我们的模型从未使用过的类型。将它们加入特征集，是提升预测力**成本最低、效果最明确**的路径。

### #4 🎯 统一到 5 日预测周期

日频预测在 A 股面临 T+1 执行错配和交易成本侵蚀两大障碍。5 日周期：
- 降低了交易频率（→降低成本）
- 避免了 T+1 日内不可卖出的尴尬
- 与模拟盘系统一致
- 每周只需 1 次决策，降低了执行复杂度

### #5 📐 精简特征集

基于 IC 分析结果，将特征集从 100+ 降至 15-25，优先保留：
- IC 绝对值 > 0.03 的特征
- 跨股票方向一致的特征
- 非价格来源的独立信号（美股、北向资金、估值）

## 总结

这个项目的**代码框架和工程意识超过了多数个人量化项目**——数据泄露防护、时间序列交叉验证、多维度评估体系的设计都有专业水准。

但项目存在一个**根本性的未解答问题**：模型到底有没有预测能力？在回答这个问题之前，所有的特征工程、模型优化、进化算法都是在装修一栋没有地基的房子。

**建议的行动顺序**：
1. 跑回测 → 看结果 → 诚实面对
2. 启动每日预测跟踪 → 积累统计样本
3. 如果回测结果显示无 alpha → 接入 investoday 独立信号源
4. 如果回测结果显示有微弱 alpha → 统一到 5 日周期 + 建立风控
5. 无论结果如何 → 简化代码，去掉冗余的回测引擎和未使用的模型

**当前最不应该做的**：继续增加模型复杂度（加新模型、加新特征变换、调超参数），在没证明基础信号有效之前，这些都是在拟合噪声。
